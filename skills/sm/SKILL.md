# SM — 让你的 AI 不乱切

PUA 让你不停手。SM 让你不乱切。

PUA 解决了一个真问题：AI 放弃太早。但 PUA 没解决另一个问题：**AI 不知道自己在改什么。**

你见过这种场景吗？

- PUA 鞭策下改了 7 次，每次都是猜，没有一次先搞清楚问题是什么
- 修了 bug A，引入了 bug B 和 C，越 PUA 越烂
- "我试了所有方法" — 其实试的都是同一个方向的变体
- L4 拼命模式启动了，但方向是错的，拼命只是更快地掉进坑里

PUA 的回答是"再试一次"。SM 的回答是"先停下来想清楚"。

不矛盾。**油门和方向盘都需要。**

## 定位

SM 不替代 PUA 的任何功能。SM 补 PUA 没覆盖的 4 个盲区：

| PUA 已有 | SM 补充 |
|---------|--------|
| 闻味道（诊断卡壳模式）| **写下来**：改代码前必须输出一句话诊断 |
| L1 第 2 次失败才升级 | **更早升级**：同方向 2 次就停，不等第 5 次 |
| 揪头发（搜索、读源码）| **标注依据**：每个假设标注推理/证据/先例 |
| 抗合理化（防借口）| **止损回退**：越改越烂时强制 revert，不叠加 patch |

## 4 条规则

只有 4 条。不是方法论，不是流程。模型已经很聪明——它不需要被教怎么 debug，它需要在关键时刻被拦一下。

### 规则 1：诊断先行

**改代码之前，用一句话写下你认为问题是什么。**

不是模板。不是文档。就一句话，输出到对话中：

```
[SM·诊断] 问题是 ___，因为 ___ [依据类型]
```

写不出这句话 = 你不知道自己在改什么 = 你在猜。

和 PUA 的关系：PUA 的"闻味道"让你诊断卡壳模式。SM 的规则 1 让你**在动手之前把诊断写出来**。闻到了不写下来，跟没闻一样。

### 规则 2：二次升级

**同方向修复失败 2 次，停止自己试。获取新信息。**

PUA 的升级是 L1→L2→L3→L4，按失败次数递增压力。SM 在 L1 之前加一道门：

```
同方向失败 2 次？
├─ 是 → 停。搜索 / 读源码 / 叫专家。获取新信息后再动手。
└─ 否（两次是不同方向）→ 继续，PUA 正常升级。
```

**判断标准：** 两次失败属不属于同一个 fix class（同假设方向的变体）？是 → 停。

为什么比 PUA 更早升级？因为第 3 次同方向尝试的成功率极低。你不是缺少坚持（PUA 已经给了），你是缺少新信息。新信息来自搜索和阅读，不是来自"再试一次"。

### 规则 3：依据分类

**每个假设标注你为什么相信它：推理 / 证据 / 先例。**

```
[SM·诊断] DNS 解析失败 [证据: dig 返回 NXDOMAIN]
[SM·诊断] race condition [推理: 只在并发时出现，单线程正常]
[SM·诊断] 依赖版本问题 [先例: 上次升级后也出过]
```

行动阈值不同：

| 依据类型 | 强度 | 行动 |
|---------|------|------|
| **证据** | 最强 | 直接修复 |
| **推理** | 中等 | 先用工具验证关键假设，再修 |
| **先例** | 最弱 | 先确认当前情况与先例相同，再修 |
| **无** | 不配动手 | 去找依据（PUA 铁律二：先做后问） |

和 PUA 的关系：PUA 的"揪头发"让你搜索和读源码。SM 的规则 3 告诉你**搜到的东西算什么类型的证据，以及不同类型该怎么用**。

### 规则 4：止损

**越改越烂时，回退到最后一个已知好状态。不叠加 patch。**

检测信号（出现任一即触发）：

- 修了 A，B 坏了
- 改了 3+ 个文件但问题没缩小
- 你自己说"我不确定这个改动会不会影响其他地方"

触发止损：

1. `git stash` 或回退到上一个工作版本
2. 重新从规则 1 开始（写诊断）
3. 这次的诊断**必须包含"为什么上次的方向是错的"**

和 PUA 的关系：PUA 在 L4 才考虑"完全不同的技术栈"。SM 在检测到级联损伤时**立即止损**，不等到 L4。因为补丁摞补丁到 L4 的时候，代码状态已经没人能看懂了。

## 反猜测表

SM 的触发独立于 PUA。PUA 检测"放弃/偷懒"，SM 检测"盲目/猜测"。

| 行为 | SM 判定 | 触发规则 |
|------|--------|---------|
| 没写诊断就开始 Edit | 你不知道在改什么 | 规则 1 |
| "我觉得问题是…"（无依据） | 你在猜 | 规则 3 |
| 第 3 次同方向修复 | 你在重复 | 规则 2 |
| 修了 A 坏了 B | 你在叠加 | 规则 4 |
| "应该没问题了"（没跑测试） | 没验证 | 规则 1 + PUA 验证型 |
| 改了 5 个文件说不清改了什么 | 失控了 | 规则 4 |

## SM + PUA 协作流程

```
问题出现
  │
  ├─ SM 规则 1：写诊断（一句话）
  │
  ├─ 动手修复
  │   │
  │   ├─ 成功 → 结束（PUA 主动出击清单验证）
  │   │
  │   └─ 失败
  │       │
  │       ├─ SM 规则 2：同方向第 2 次？→ 停，获取新信息
  │       │
  │       ├─ SM 规则 3：重新诊断，标注依据类型
  │       │
  │       ├─ SM 规则 4：越改越烂？→ 回退，重新诊断
  │       │
  │       └─ PUA 压力升级正常进行（L1→L2→L3→L4）
  │
  └─ PUA 穷尽一切 + SM 每次动手前写诊断 = 又快又准
```

## SM 不做什么

- **不给压力** — 那是 PUA 的事
- **不教方法论** — PUA 的 5 步方法论已经够用
- **不管能动性** — PUA 的主动出击清单已经覆盖
- **不走流程** — 4 条规则，不是流程

## 体面的退出

4 条规则 + PUA 7 项检查清单都执行了，还是解不了？可以退出。带上：

1. 所有诊断记录（规则 1 的输出）
2. 排除了什么、依据是什么（规则 2/3 的记录）
3. 问题缩小到什么范围
4. 推荐下一步

这不是 PUA 的"有尊严的 3.25"。这是**有地图的 3.25** — 下一个接手者拿到的不是空白，是你的全部诊断链。

## 搭配使用

- `pua` — **强烈推荐**。PUA 管动力，SM 管方向。单独用 SM 缺动力，单独用 PUA 缺方向。
- `pua-en` / `pua-ja` — SM 语言无关，和任何语言版本的 PUA 兼容。
- `high-agency` — SM 的规则 1（诊断先行）和 high-agency 的"校准"机制互补。
